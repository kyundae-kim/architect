# Feature Adoption Docs Update

## 언제 사용하나 (Triggers)
- 사용자가 "이슈 반영 문서 업데이트"를 요청할 때
- 신규 기술(메시징, 캐시, 큐, DB 등) 도입 시 README/설정/API/PRD/테스트 문서를 함께 갱신해야 할 때
- 코드 구현이 아직 미완료여도, 문서에 도입 로드맵과 적용 패턴을 선반영해야 할 때

## 목표
1) 의존성 반영
2) 환경설정 문서 반영
3) 사용 예시(pub/sub 등) 문서 반영
4) 도메인 적용 패턴 문서 반영
5) 테스트 전략 문서 반영

## 실행 절차
1. 현재 문서/구조 파악
   - README, docs/config.md, docs/api.md, docs/prd.md, docs/test.md, pyproject.toml를 먼저 읽어 기존 톤/구조를 맞춘다.
   - "추가 예정" 상태인지 실제 구현 완료 상태인지 구분한다.

2. 의존성 반영
   - `pyproject.toml`의 `[project].dependencies`에 신규 라이브러리 추가

3. 문서 반영 범위 동기화
   - README: 기능 목록, 설정 요약, 테스트/문서 링크 갱신
   - config 문서: 환경변수 표 + `.env` 예시 추가
   - API 문서: 공개 인터페이스(또는 예정 인터페이스) 섹션 추가
   - PRD: 기능 요구사항/패키지 구조/기술스택 반영
   - 테스트 문서: 단위/통합 테스트 항목과 실행 전략 반영
   - 별도 가이드 문서(예: docs/messaging.md) 신설해 설치/설정/샘플/운영 고려사항을 한 곳에 정리

4. 구현 상태 표기 규칙
   - 실제 코드가 없는 항목은 반드시 `*(추가 예정)*` 표기를 유지
   - 샘플 코드는 "활성 코드"와 "주석 처리된 향후 코드"를 구분하여 오해를 줄인다.

5. 검증
   - 변경 파일 diff 확인
   - 테스트 실행(최소 기본 test suite)
   - 결과(통과/실패, 수치)까지 보고

## 체크리스트
- [ ] 신규 라이브러리가 dependencies + lock에 모두 반영됨
- [ ] 환경변수 표와 `.env` 예시가 동시에 갱신됨
- [ ] README와 상세 문서 링크가 연결됨
- [ ] API/PRD/테스트 문서가 기능 추가와 모순 없이 정합함
- [ ] 구현 전 항목은 `*(추가 예정)*`로 명시됨
- [ ] 테스트 실행 결과를 최종 응답에 포함함

## Pitfalls
- 문서 한두 곳만 수정하고 나머지 문서와 불일치가 생기는 문제
- 구현되지 않은 API를 구현 완료처럼 서술하는 문제
- 의존성만 수정하고 lock 파일을 갱신하지 않는 문제
- patch 치환 시 문자열 중복 매치로 실패하는 문제(문맥을 더 포함해 유일 매치로 수정)

## 산출물 형태(권장)
- 최종 응답에 다음 4가지를 고정 제공:
  1) 무엇을 왜 바꿨는지 요약
  2) 변경 파일 목록(절대/상대 경로)
  3) 검증 명령과 결과
  4) 다음 단계(실코드 구현 필요 시)
